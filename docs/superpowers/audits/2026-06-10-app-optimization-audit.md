## Executive summary

Eight cross-cutting themes:

1. **The app has never met production conditions, and it shows in the defaults.** Everything runs against local Supabase; there is no deploy pipeline, no hosted project, no environment-promotion runbook (Critic #1). Consequences are already latent in the repo: Sentry source maps have never uploaded in CI (C39), browser Sentry events will die to ad-blockers with no tunnel (C51), sign-out redirects to literal `localhost:3000` in production (C50), and the web app ships zero security headers (C47).

2. **The upcoming mobile device tests will trip on basics before perf is even visible.** Home/officials/settings screens cannot scroll on native (C8/U0); completing or skipping calibration bounces the user straight back to /calibrate until app restart (U1 — hand-confirmed); cold start is default-splash → spinner → app with no splash/icon config (C14); the officials list renders black-on-dark text in dark mode (U3); mobile sign-up success renders as a red error banner (U6).

3. **`districts_geojson` is the single biggest byte win in the codebase.** Three dimensions independently converged on it (C1, C9, C16): full-resolution 9-decimal TIGER MultiPolygons — multi-MB for coastal states — are fetched client-side on every home visit on both platforms, with the statewide senate polygon downloaded twice (S1+S2 share geometry). A display-tolerance `ST_SimplifyPreserveTopology` + 5-decimal precision in the view is a patch-tier fix with ~90%+ payload reduction.

4. **Detail pages are network-chatty by composition.** A federal detail render fires ~29 PostgREST round-trips (state ~22) all client-side after hydration with zero SSR prefetch (C18, C4); `fetchOfficialMissedVotes` downloads every roll call of the congress per page view (C17); collapsed-by-default subsections eagerly fetch full payloads even though the hooks already expose `enabled` (C12); middleware adds 2 more sequential round-trips per navigation (C3/C49).

5. **Ingest has silent data-quality failures, not just freshness risk.** The state mobilize town-halls adapter writes 0 rows — every event dropped at the upsert layer (C31); the Senate PTR/FD adapters call a stub search that always returns `[]`, so `seed:federal-ptrs`/`seed:federal-fds` silently ingest House data only (C32); district-offices re-runs duplicate all ~450 office rows (C33); name-based official matching picks an arbitrary row on ambiguity with no detection — the most reputationally dangerous failure mode the product has (Critic #3).

6. **Pre-launch compliance basics are missing.** No account-deletion path exists, which hard-blocks App Store submission under Guideline 5.1.1(v) independent of GDPR (Critic #2); the privacy-policy placeholder is factually wrong in both directions; sign-up has no ToS/consent; next@15.5.15 carries HIGH advisories including an App Router middleware-bypass class that directly targets Chiaro's middleware-enforced auth (U16 — hand-confirmed via `pnpm audit`); password policy is 6 chars server-side vs 8 client-side (C54/U18); the GeocodIO spend happens before the rate-limit check (U17 — hand-confirmed).

7. **CI is ~2× slower than necessary and the toolchain has gaps.** The 4-job graph is fully serialized with zero data dependencies — ~18 min wall vs ~9 achievable (C40); zero lint/format tooling in a 12-package monorepo (C41); no Windows CI despite Windows being the primary dev environment (C42); `supabase/setup-cli@latest` despite a documented hang incident (C43); no E2E coverage of any user-critical flow even though CI already has every prerequisite for a minimal Playwright job (U8).

8. **Dead and duplicated UI architecture is taxing every slice.** The slice-4-era MetricCardShell category-card subsystem has zero production consumers yet keeps absorbing reskin/migration work (C24); the 12 federal/state detail cards hand-roll the same 3-branch shell and have already drifted user-visibly (C25); legacy COLORS is fully dead but CLAUDE.md still instructs new code to use it (C28); `push_tokens` is a schema-only orphan with no app consumers (Critic #6).

#### C1 · HIGH · patch — Home page downloads full-resolution TIGER district geometry (incl. the statewide senate polygon twice) through the un-simplified districts_geojson view

**Claim.** getMyDistricts fetches the geometry column for every user district from the districts_geojson view, which is defined as ST_AsGeoJSON(geometry::geometry) with no simplification and default 9-decimal precision. A calibrated user has ~5 districts: 1 federal house + 2 senate rows (S1 + S2 share the SAME statewide geometry per Gotcha #2 / tiger-config.ts senate dual-row design — so the full state boundary is downloaded twice) + state SS + SH. Full-resolution TIGER statewide MultiPolygons as 9-decimal GeoJSON are routinely multi-megabyte (coastline states worst). This payload is fetched client-side after hydration on every home visit (5-min staleTime, 30-min gcTime — re-paid on most sessions), then parsed and handed to Leaflet. It is very likely the dominant cost of the home page, dwarfing the JS bundle.

**Evidence.** packages/db/supabase/migrations/0007_districts_geojson_view.sql: `ST_AsGeoJSON(geometry::geometry)::jsonb as geometry` — no ST_SimplifyPreserveTopology, no maxdecimaldigits arg. packages/location/src/queries.ts:61-66 selects 'id, tier, state, code, name, geometry' from districts_geojson for all user district ids. apps/web/components/DistrictPanel.tsx:25,41-47 passes raw geometry into DistrictMap. CLAUDE.md Gotcha #2: senate is two rows per state with the same geometry. packages/location/src/hooks.ts:13-14 (5-min staleTime).

**Recommendation.** Append-only migration replacing the view body with display-appropriate geometry: ST_AsGeoJSON(ST_SimplifyPreserveTopology(geometry::geometry, ~0.001), 5) — 5-decimal precision (~1 m) alone cuts ~40% of coordinate bytes; topology-preserving simplification at map-display tolerance typically cuts 90%+. Keep the raw column available under a different name if anything needs exactness (calibration uses server-side ST_Contains, not this view). Additionally, dedupe the senate geometry client-side: fetch geometry once per unique (state, 'S1') and reuse for S2, or filter one senate row out of the map fetch in getMyDistricts.

**Adversarial verifier.** All links in the chain verified. migrations/0007_districts_geojson_view.sql: ST_AsGeoJSON(geometry::geometry)::jsonb with no ST_SimplifyPreserveTopology and no maxdecimaldigits arg (defaults to 9); grep of migrations shows no later migration replaces the view, and grep of the whole seed tree shows zero simplification at ingest — districts store full-resolution TIGER geometry. packages/location/src/queries.ts:61-66 getMyDistricts selects geometry for ALL user district ids from districts_geojson. Senate duplication confirmed end-to-end: calibrate-location/geocodio.ts:60-64 pushes BOTH ${state}-S1 and ${state}-S2, and apply_calibration (0008) inserts every matched row — so the statewide polygon (same geometry per Gotcha #2) is linked twice and downloaded twice. DistrictPanel.tsx:41-47 passes raw geometry into the dynamic Leaflet map; hooks.ts:13-14 confirms 5-min staleTime / 30-min gcTime (in-memory only, so every hard load re-pays). Severity high is fair — full-res 9-decimal statewide TIGER MultiPolygons are routinely multi-MB and this is client-fetched on the home page. Recommendation fits conventions (append-only migration; create-or-replace keeps the jsonb column type; calibration containment is GeocodIO-side, not this view, so display-tolerance simplification is safe) and the fix also benefits mobile DistrictMap, which consumes the same getMyDistricts.
