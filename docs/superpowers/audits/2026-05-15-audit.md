## Top 5 (do these first)

1. **Remove tracked debug/mockup junk from root** — `server.js`, `chiaro-demo-report.html`, the 9 top-level UI mockup folders (`Bill_Detail/`, `Home_Screen/`, `Senator_Screen/`, etc.), and two empty `_tmp_6_*` files are all tracked. They confuse the file tree, pollute `git status`, and the mockups aren't wired into anything. The root `supabase/` stub (only `snippets/`) is in this same bucket — delete it; it's the "trap" that breaks `supabase start` when run from repo root. (S, Low risk to defer but high signal-to-noise win.)
2. **Promote `DISTRICT_GROUPS` + tier label/color maps to `packages/location`** — currently duplicated across `apps/web/components/DistrictPanel.tsx:26`, `apps/mobile/components/DistrictPanel.tsx:18`, `apps/web/components/DistrictMap.tsx:34`, `apps/mobile/components/DistrictMap.tsx:33`. Slice 3 will need a 5th copy. (S)
3. **Make `calibrate-location` writes transactional via a SECURITY DEFINER RPC** — the delete→upsert→insert sequence in `functions/calibrate-location/index.ts:84-129` can leave a user with an updated `user_locations` row and zero `user_districts` rows. Move all three writes into one Postgres function; Edge Function just calls `admin.rpc('apply_calibration', ...)`. (M, Med-High risk)
4. **TIGER ingest needs retry + idempotent resume** — `seed/tiger-ingest.ts:25-41` aborts on first 404. With 51 per-state shapefile URLs each pull, a 1.5% transient-failure rate means most runs fail. Add per-URL retry with exponential backoff and skip already-ingested `(tier, code, source_version)` rows. (M, High risk — gates first prod ingest.)
5. **Lift the `as unknown as ChiaroClient` casts** — `apps/web/lib/supabase/{client,server}.ts:9,32` and the `as never` casts in `DistrictPanel.tsx` are bandages for a `@supabase/ssr`@0.5.2 ↔ `supabase-js`@2.105.3 generic-arity gap. Latest `@supabase/ssr` (0.6+) was recompiled against the 5-generic client. Bump `^0.5.0` → `^0.6.0` and remove all four casts; verify the type passes. (S, Low.)

## B. Performance / reliability

- **`calibrate-location` write sequence is non-atomic** — see Top 5 #3. The current code's comment at `index.ts:78-82` explicitly punts; an RPC fixes it cleanly. Effort M, risk High.
- **TIGER ingest single-shot failure** — Top 5 #4. Effort M, risk High.
- **N+1 in calibrate-location district lookup** — `index.ts:106-128` issues a separate `select` per resolved district (typically 6–10 per user). Replace with one `.in('code', ...)` query keyed on `(tier, code)` tuples, or push the whole resolve into the RPC from finding #3. Effort S, risk Low.
- **Middleware probe runs on every page request** — `apps/web/middleware.ts:41-44` issues a HEAD `select count` for *every* authenticated request not in the allow-list. Even at "head + exact count" this is a network round-trip per nav. Cache the boolean in a cookie set by `/calibrate` (and cleared on sign-out). Effort S, risk Low.
- **Mobile `_layout.tsx` recheck runs once per mount, no debounce / no reactivity to a successful calibrate** — `apps/mobile/app/(app)/_layout.tsx:12-32` runs once on mount; if the user calibrates, the layout doesn't recompute until next mount. Subscribe to a focus event or use Supabase realtime on `user_locations`. Effort S.
- **`DistrictPanel` (both apps) double-fetches user** — calls `getMyDistricts` *and* a separate `supabase.auth.getUser()` then a second select on `user_locations`. Combine into one `getMyLocationAndDistricts` helper. Effort S.
- **`districts.geometry` GiST index exists (migration 0003:23) — good.** No PostGIS index covers the *combined* tier+state common filter on `user_districts`; the `(district_id)` and `(tier)` indexes both exist but a covering `(user_id, tier)` would speed officials-grouped-by-tier queries in slice 3. Effort S.
- **Map polygon rendering on mobile flattens MultiPolygon coords ring-by-ring with no simplification** — `apps/mobile/components/DistrictMap.tsx:110-115`. A US-House district can be 20–80 KB of coordinates. At 6+ overlapping polygons on a phone, this stutters. Add a server-side `ST_Simplify` for low-zoom views or precompute a simplified tippecanoe/MVT tile set. Effort M for the long-term fix; S for a `ST_SimplifyPreserveTopology` view as an interim. Risk Med.
- **`max_rows = 1000` in `config.toml:18` will start truncating** when slice 3's officials query joins by district — a single user has ~10 districts, but listing officials across an entire state could hit the cap. Plan to set per-table or filter explicitly. Low risk now, Med once officials lands.

## C. Security / secrets

- **No leaked secrets in tracked files.** GEOCODIO_KEY value `32523063...` is not present in the repo (confirmed via grep) — only the placeholder in `.env.example`. The leak was in chat history, not in committed code.
- **`calibrate-location` has no rate limit.** A signed-in user can hammer GeocodIO (2,500/day free tier — burns through fast on abuse). `config.toml [auth.rate_limit]` doesn't cover Edge Functions. Add per-user throttle: insert a `last_calibrated_at` check in the SECURITY DEFINER RPC (e.g., reject if < 60s since last). Effort S, risk Med.
- **`profiles_select_authenticated` allows any authenticated user to read every profile** — `0002_profiles_rls.sql:3-6`. Fine for slice 1 (only `display_name`/`username` are exposed) but as soon as slice 3+ adds public-figure profile fields or PII, this needs revisiting. Add a doc-comment flagging the constraint. Risk Low now.
- **`SUPABASE_SERVICE_ROLE_KEY` in Edge Function is correctly server-only** — confirmed at `index.ts:7`; only Deno worker can read it. Good.
- **Mobile + web both write `chiaro_skip_calibrate` without `Secure`/`SameSite` flags** — `apps/web/app/calibrate/page.tsx:41` sets `document.cookie='chiaro_skip_calibrate=1; path=/'` plain. Cross-site script could set this and bypass the calibration gate (not a security boundary, just UX, but worth `SameSite=Lax`). Effort S.
- **No CAPTCHA / abuse path for sign-up.** `config.toml:208` has captcha commented out. Geocode-on-signup is one extension away from being weaponizable. Risk Med once deployed publicly.

## D. Infrastructure / CI / build

- **CI jobs `db`, `functions`, `test` each `supabase start` independently** — `.github/workflows/ci.yml:23,78,106`. Booting Supabase locally takes ~45-90s × 3 = ~3-4 min of wasted CI. Either combine `functions`+`test` (both need running Supabase + edge function) into one job, or use `supabase setup --cache` once and reuse. Effort S, risk Low.
- **TIGER seed re-runs in two separate CI jobs** (`db` and `test`). The TIGER cache key is shared, but the actual `db:seed-tiger` Node script still runs the ingest each time (~60-180s × 2). Materialize a `pg_dump` into the cache, restore in subsequent jobs. Effort M, risk Med.
- **No branch protection signals in repo** — workflow doesn't fail PRs missing `GEOCODIO_KEY` secret cleanly; live-Geocodio test in `test` job just skips silently if secret absent. Add an explicit `if: secrets.GEOCODIO_KEY != ''` guard with a `continue-on-error: false` matching job. Effort S.
- **No lint/format step in CI** — no ESLint, Prettier, or Biome config visible. As packages multiply this will bite. Effort S, risk Med.
- **`pnpm db:check-types-drift` is fine, but `pnpm db:gen-types` outputs to `packages/db/src/types.ts` which is then diffed against `git`** — works, but generates the file as a side effect of CI. If a developer runs the check locally they dirty their tree. Wrap in `--check` mode or write to `/tmp` first. Effort S.
- **Mobile DoD (slice 2.5) gating signal missing** — no CI job builds the Expo bundle (`expo export`) or runs `tsc` on mobile (mobile typecheck is in `pnpm typecheck` via Turbo, but no smoke). Add an EAS Build Cloud workflow trigger gated to `main` for the dev profile. Effort M.

## F. Architecture / forward-looking

- **Introduce a `packages/api` (or `packages/queries`) layer** before Slice 3. Today both apps call `supabase.from(...)` inline in components. Centralize per-resource: `getMyProfile`, `getMyDistricts`, `getOfficialsForMyDistricts`, `getBillsForOfficial`, etc., returning typed shapes. Lets you swap to a server-side BFF later if needed.
- **Two profile types (citizen vs public-figure) will force a schema decision now.** Options: (a) single `profiles` table + a `profile_kind` enum + nullable public-figure-specific columns; (b) `profiles` + `official_profiles` 1-to-1 with FK to `officials`. Option (b) is cleaner for RLS — citizen-profile RLS is "select_self_update_self", officials need "select_all, no_user_update". Decide before slice 3 schema lands. Effort: design-only S, schema M.
- **GeoJSON view pattern should generalize.** `districts_geojson` (migration 0007) papers over PostgREST's WKB-hex default for one table. The `user_locations` query in `packages/location/src/queries.ts:12-19` still selects raw `home_location` and discards it. Add a `user_locations_geojson` view in slice 3 (or whenever you touch `user_locations`); document the pattern in `docs/`.
- **Realtime for `votes` is foreseeable** — Slice 1 disabled realtime per app, but votes-as-they-happen will want it. Plan the channel/topic shape (per-bill? per-official?) before schema.
- **Push notifications for bill votes need an Expo push token table now.** Trivial schema (`push_tokens(user_id, token, platform)`), but easier to land in slice 3's already-large surface than as a one-off later.
- **`packages/db/src/index.ts` is the canonical DB-types export.** Today it re-exports `Database` from `types.ts` only. As migrations grow, add helper unions (e.g., `type DistrictTier = Database['public']['Enums']['district_tier']`) so app code doesn't re-declare enums (`packages/location/src/types.ts:1-10` does this today).
- **Root `supabase/` stub directory should be deleted.** It contains only `snippets/` and is the "trap" — `supabase start` from root spins wrong-named containers. Remove the directory; if `snippets/` has anything in it, move under `packages/db/supabase/snippets/`. Effort S, risk Low.
