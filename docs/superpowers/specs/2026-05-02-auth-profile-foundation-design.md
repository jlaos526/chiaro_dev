# Auth + Profile Foundation — Design

**Date:** 2026-05-02
**Status:** Draft for review
**Slice:** Foundational vertical slice — auth and profile creation, end-to-end on mobile and web, with Supabase RLS enforced.

## Goal

Stand up the smallest end-to-end slice of Chiaro that proves the four pillars of the stack work together:

1. A user can sign up on mobile (Expo) or web (Next.js).
2. On signup, a Postgres trigger materializes a stub `profiles` row.
3. The user can complete their profile from either app.
4. Row-Level Security enforces that one user cannot modify another user's profile.

No public-figure profiles, no location features, no Storage uploads. The only purpose of this slice is to lock in the auth, schema, RLS, and shared-package patterns that every later slice will build on.

## Decisions

| # | Decision | Choice |
|---|---|---|
| 1 | Auth method | Email + password (Supabase Auth) |
| 2 | Profile-row creation | Postgres trigger creates a stub row on `auth.users` insert; client `UPDATE`s it |
| 3 | Public-figure profile schema | Out of scope — separate `public_figures` table will be added in a later slice |
| 4 | Profile fields | `id`, `display_name`, `username` (citext, lowercase, regex-validated), `completed`, `created_at`, `updated_at` |
| 5 | Completion gate | `completed` flag exists, set to `true` on form submit, but is not yet enforced as a precondition for any feature |
| 6 | Monorepo split | `apps/{mobile,web}` + `packages/{db, supabase-client, profile}` |
| 7 | RLS test strategy | pgTAP for policy logic + node integration test for client → PostgREST → RLS path |

## Database

### Migration `0001_profiles.sql`

```sql
create extension if not exists citext;

create table public.profiles (
  id            uuid        primary key references auth.users(id) on delete cascade,
  display_name  text                 check (display_name is null or char_length(display_name) between 1 and 50),
  username      citext      unique   check (username is null or username ~ '^[a-z0-9_]{3,20}$'),
  completed     boolean     not null default false,
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now()
);

create or replace function public.handle_new_user()
returns trigger
language plpgsql
security definer
set search_path = public
as $$
begin
  insert into public.profiles (id) values (new.id);
  return new;
end;
$$;

create trigger on_auth_user_created
after insert on auth.users
for each row execute function public.handle_new_user();

create or replace function public.touch_updated_at()
returns trigger language plpgsql as $$
begin new.updated_at := now(); return new; end;
$$;

create trigger profiles_touch_updated_at
before update on public.profiles
for each row execute function public.touch_updated_at();
```

### Migration `0002_profiles_rls.sql`

```sql
alter table public.profiles enable row level security;

create policy "profiles_select_authenticated"
  on public.profiles for select
  to authenticated
  using (true);

create policy "profiles_update_self"
  on public.profiles for update
  to authenticated
  using      (id = (select auth.uid()))
  with check (id = (select auth.uid()));

revoke update on public.profiles from authenticated;
grant  update (display_name, username, completed) on public.profiles to authenticated;
```

### Schema rationale

- **Trigger pattern.** Every signup produces exactly one `profiles` row. There is no path to an "auth user without a profile" state. The trigger runs as `security definer`, scoped narrowly to a single hardcoded insert.
- **No insert policy.** Only the trigger writes new rows. This eliminates a class of "wrong id on insert" bugs.
- **Column-level grants.** Even with the row-level update policy, an attacker submitting `id = <other>` or `created_at = <past>` is harmless — those columns aren't grantable to `authenticated`.
- **`(select auth.uid())` not `auth.uid()`.** Postgres can cache the subquery result per query; this is the documented Supabase performance pattern.
- **`citext` username with lowercase-only regex.** Uniqueness is case-insensitive; storage is canonically lowercase. The form normalizes input on submit.
- **`completed` is user-writable.** The client flips it to `true` on form success. There is no DB-level enforcement of "completed implies fields filled" yet — that constraint will be added in the slice that introduces a feature requiring it.
- **Cascade delete.** `on delete cascade` from `auth.users` removes the profile when the account is deleted via Supabase Auth.

## Client Wiring

### `packages/supabase-client`

```ts
// client.ts
import { createClient, SupabaseClient } from '@supabase/supabase-js'
import type { Database } from '@chiaro/db'

export type ChiaroClient = SupabaseClient<Database>

export function createChiaroClient(opts: {
  url: string
  anonKey: string
  storage?: SupportedStorage
}): ChiaroClient {
  return createClient<Database>(opts.url, opts.anonKey, {
    auth: {
      storage: opts.storage,
      persistSession: true,
      autoRefreshToken: true,
      detectSessionInUrl: false,
    },
  })
}
```

`auth.ts` exports thin wrappers — `signUp`, `signIn`, `signOut`, `getSession` — that delegate to `client.auth.*` and surface typed errors.

### Mobile (`apps/mobile`)

- One client instance in `lib/supabase.ts` with `storage: AsyncStorage` from `@react-native-async-storage/async-storage`.
- `expo-router` layout structure:
  - `app/(auth)/sign-in.tsx`, `app/(auth)/sign-up.tsx` — public.
  - `app/(app)/_layout.tsx` — guard; redirects to `/sign-in` if no session.
  - `app/(app)/index.tsx` — home; renders `display_name` or "Complete your profile" CTA.
  - `app/(app)/profile/edit.tsx` — the profile form.
- Session is read once at launch via `supabase.auth.getSession()`; an `onAuthStateChange` listener keeps the auth context in sync. An `AppState` listener calls `startAutoRefresh()` / `stopAutoRefresh()` on foreground/background, which is the documented Supabase + RN refresh pattern.

### Web (`apps/web`)

- Two factories from `@supabase/ssr`:
  - `createServerClient()` — uses Next's `cookies()`; consumed in Server Components, Route Handlers, and middleware.
  - `createBrowserClient()` — used in client components for mutations and `onAuthStateChange`.
- `middleware.ts` refreshes the session cookie on every request using the Supabase SSR pattern. The matcher is copied verbatim from the official Supabase Next.js example.
- Pages:
  - `/sign-in`, `/sign-up` — public; client components for forms.
  - `/` — server component; reads session and profile, renders home or redirects to `/sign-in`.
  - `/profile/edit` — client component with the form.

### Profile feature module (`packages/profile`)

```ts
// schema.ts
export const profileFormSchema = z.object({
  display_name: z.string().trim().min(1).max(50),
  username: z.string().trim().toLowerCase().regex(/^[a-z0-9_]{3,20}$/),
})
export type ProfileFormInput = z.infer<typeof profileFormSchema>

// mutations.ts
export async function updateMyProfile(client: ChiaroClient, input: ProfileFormInput) {
  const { data: { user } } = await client.auth.getUser()
  if (!user) throw new Error('Not signed in')
  const { data, error } = await client
    .from('profiles')
    .update({ ...input, completed: true })
    .eq('id', user.id)
    .select()
    .single()
  if (error) throw mapProfileError(error)  // 23505 → "Username taken"
  return data
}

// queries.ts — getMyProfile() likewise
```

`mapProfileError` is a thin Postgres-error → human-message helper local to this package.

### End-to-end flow (mobile and web, identical contract)

1. User submits sign-up form (`email`, `password`).
2. `supabase.auth.signUp(...)` creates `auth.users` row.
3. `on_auth_user_created` trigger fires → `profiles` row created with `completed=false`, all other fields null.
4. App reads session, navigates to home.
5. Home queries `getMyProfile()`; if `completed === false`, renders "Complete your profile" CTA.
6. User opens `/profile/edit`, submits `{display_name, username}`.
7. `updateMyProfile()` issues `UPDATE profiles SET display_name=$1, username=$2, completed=true WHERE id=auth.uid()`.
8. RLS allows the update; column grants restrict the writable set.
9. On 23505, the form surfaces "Username taken." On success, the form navigates back to home.

## Testing

### Layer 1 — pgTAP (`packages/db/supabase/tests/profiles_rls.test.sql`)

Run via `supabase test db` against a fresh local instance. Asserts:

| # | Test | Expected |
|---|---|---|
| 1 | Trigger fires on `auth.users` insert | New `profiles` row exists with `completed=false`, `display_name=null`, `username=null` |
| 2 | User A updates own row (display_name, username, completed) | Succeeds; `updated_at` advances |
| 3 | User A updates user B's row | 0 rows affected (RLS filter); no error raised |
| 4 | User A attempts `update profiles set id=<B>` | Permission denied (column grant) |
| 5 | User A attempts `update profiles set created_at=...` | Permission denied (column grant) |
| 6 | Anon role `select * from profiles` | 0 rows returned (RLS enabled, no `to anon` policy — anon retains table-level grant from Supabase defaults, but no row matches) |
| 7 | Authenticated role `select` of any row | Succeeds |
| 8 | `delete from auth.users where id=<A>` | Profile row gone (cascade) |
| 9 | Username uniqueness | Second insert with same username (different case) fails with 23505 |

### Layer 2 — Node integration (`packages/profile/test/integration.test.ts`)

Vitest, drives the actual `@supabase/supabase-js` client against a local Supabase instance booted in CI:

| # | Test | What it adds over pgTAP |
|---|---|---|
| 1 | `signUp` then `getMyProfile()` returns stub row | Trigger + client query path together (catches missing `select` grant) |
| 2 | `updateMyProfile({display_name, username})` succeeds, `completed=true` | Update flows through `packages/supabase-client` and column grants are correctly wired |
| 3 | Sign in as user A, attempt to update user B's row | `data` is empty; client surfaces this without throwing |
| 4 | Username conflict → "Username taken" | 23505 → human-message mapping is correct |
| 5 | Anonymous client calling `updateMyProfile` | Throws "Not signed in" before hitting DB |

### Layer 3 — App-level e2e

**Out of scope.** No Detox or Playwright in this slice. The data path is covered by the integration tests; rendering correctness will be added when the home screen has real content beyond a placeholder.

### CI pipeline (`.github/workflows/ci.yml`)

```
job: db
  - supabase start
  - supabase db reset                                            # applies migrations to fresh DB
  - supabase test db                                             # pgTAP suite
  - supabase gen types typescript --local > packages/db/src/types.ts
  - git diff --exit-code packages/db/src/types.ts                # fails if generated types are stale

job: build (depends-on: db)
  - pnpm -r build
  - pnpm -r typecheck

job: test (depends-on: db)
  - supabase start
  - pnpm -r test                                                 # vitest in packages/profile
```

The `git diff --exit-code` step on generated types ensures schema and code can't drift — if a migration changes the schema, the dev who wrote it must commit the regenerated types or CI fails.

## Risks and Known Unknowns

1. **Trigger + `security definer`.** Standard Supabase pattern but a known footgun. Mitigation: function does a single hardcoded insert with `set search_path = public`; pgTAP test #1 covers it. Do not extend without re-reviewing.
2. **Username case handling.** `citext` makes uniqueness case-insensitive; the regex enforces lowercase-only at storage. A future ALTER dropping `citext` would require a backfill — locked in for the foreseeable future.
3. **`enable_confirmations = false` in dev only.** Forgetting to enable it in production is a real risk. Mitigation: production environment config will be reviewed before first prod deploy. Not part of this slice.
4. **Web SSR cookie handling.** `@supabase/ssr` middleware must run on every request; a wrong matcher silently breaks session refresh. Mitigation: matcher copied verbatim from the official Supabase Next.js example.
5. **RN auth refresh on background → foreground.** Documented gotcha. Tested manually in this slice; no automated coverage.

## Out of Scope (Deferred)

- Public-figure profiles (table, admin UI, policies)
- Location features (PostGIS, `home_location`, district lookup)
- Avatar upload (Supabase Storage, `avatar_url`)
- OAuth providers (Apple, Google)
- Magic link auth
- Password reset UI (Supabase sends the email; the in-app reset page is a later slice)
- Email confirmation UI (a "check your email" placeholder is enough until production config arrives)
- Following officials, bills, votes — every product feature beyond auth + profile
- App-level e2e (Detox / Playwright)
- Bio and other profile fields
- Live username availability check
- In-app account deletion

## Definition of Done

- Fresh clone → `pnpm install` → `supabase start` → `pnpm dev` runs both apps locally.
- Sign up on web → stub profile visible in SQL → fill form → updated row visible.
- Sign up on mobile (Expo Go on a physical device or simulator) → same flow works.
- pgTAP suite green; integration tests green; generated types not drifted.
- A second user cannot edit the first user's profile, verified by both pgTAP and integration tests.
