# S83 — Playwright E2E foundations + a11y automation (design)

Re-planned Wave 3, fourth slice (audit U8 + U11 — U11 folded in from dissolved
S84). Full tier. The harness decisions ARE the work; the journeys are thin.

## Decisions

### D1 — Target environment: local prod build + local Supabase (both envs)
Playwright drives `next start` (production build) against the local Supabase
stack. In CI this is a NEW parallel `e2e` job (own Supabase boot, pinned CLI
2.98.1, served Edge Function with GEOCODIO_KEY per the test-job pattern).
Locally the same specs run against the developer stack. **The web build
inlines NEXT_PUBLIC_SUPABASE_URL/ANON_KEY at build time** — the e2e job
builds web AFTER `supabase status` yields the keys.

### D2 — Data strategy: idempotent fixture seed, NOT TIGER
`apps/web/e2e/seed-e2e.mjs` (service-role, reads `supabase status`) seeds:
- 4 districts CONTAINING SF City Hall (-122.4194, 37.7793): `CA-11`
  (federal_house), `CA-S1`/`CA-S2` (federal_senate), `CA-SS-11` +
  `CA-SH-17` (state tiers, TIGER `STATE-{SS|SH}-N` format per Gotcha #8) —
  each an `ON CONFLICT (tier, code) DO NOTHING` insert followed by a lookup,
  so a TIGER-seeded local DB reuses the REAL rows (no unique-violation, the
  slice-67 collision lesson) while the bare CI DB gets the fixture polygons.
- 3 federal + 2 state officials attached to those district ids (upsert by
  bioguide/openstates id) + one scorecard org/rating so alignment has data.
- `pnpm seed:issue-catalog` (real seed; no keys) for the quiz journey.

### D3 — The three journeys (`apps/web/e2e/*.spec.ts`)
1. **signup → calibrate → home**: fresh unique email per run (throttle U17
   never bites a fresh user; local confirmations are OFF so signup yields a
   session), calibrate via the S79.5 **sample-address affordance** (SF City
   Hall — matches the fixture polygons through the LIVE GeocodIO path; the
   test job already spends bounded live geocodes, precedent kept), assert
   home renders the officials card with fixture names.
2. **officials detail**: from home, click a fixture rep → `/officials/[id]`
   renders h1 + card h2s (the S80 shell contract).
3. **quiz → alignment**: `/issues` flow → topic → lens → quiz → save →
   rep detail shows the alignment strip (not the setup CTA).
Each journey captures a full-page screenshot to `e2e-artifacts/` (uploaded in
CI, 7-day retention) — the README-screenshot follow-up rides on these.

### D4 — CI wiring
New parallel `e2e` job: Supabase boot → db:reset → seed-e2e + issue-catalog →
serve calibrate-location (env-file pattern) → build web with local env →
`next start` → `playwright install chromium --with-deps` (browser dir cached
via actions/cache keyed on the @playwright/test version) → run specs.
**NOT added to branch-protection required contexts v1** — new E2E needs a
stability week before it can block merges; the checks-table merge gate covers
it meanwhile. Revisit after ~a week of runs.

### D5 — U11 vitest-axe (officials-ui)
`vitest-axe` devDep + one `test/a11y/axe.test.tsx` running axe over
representative real-DOM renders (DetailCardShell ×4 branches, EventRowList,
OfficialsList with rows, AuthForm, CalibrateScreen, SettingsScreen rows,
BrandNavRailBody, LegalBody). Trivial violations get fixed in-slice;
non-trivial ones get a documented `disableRules` entry + follow-up note —
the gate's value is preventing NEW regressions.
